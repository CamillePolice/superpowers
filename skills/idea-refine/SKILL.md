# Idea Refine

## When to Use

- Vague or ambiguous requirement
- Need to validate a direction before writing a spec
- Multiple possible approaches not yet evaluated

## Core Process

### Phase 1 — Clarification (one question at a time)

Assess whether the request is clear enough to proceed. A request is clear when you can answer all of these:
- What exactly is being built or changed?
- Who uses it and what triggers it?
- What are the success criteria?
- Are there constraints (performance, compatibility, scope)?

If **any** is unanswerable, ask one question at a time. Wait for the answer before asking the next. Prefer multiple-choice questions when possible. Stop once the request is clear enough.

**Never proceed to Phase 2 while requirements are still ambiguous.**

### Phase 2 — Explore directions

Once the goal is clear:
- Reframe as a "How Might We" problem
- Propose 2-3 directions with trade-offs and a recommendation
- Surface hidden assumptions and what could derail each direction
- Stress-test against: user value, feasibility, scope

### Phase 3 — Sharpen & confirm

Present the recommended direction and get explicit approval before producing any plan or spec.

Once approved, produce a short one-pager:
- Problem statement
- Chosen direction and rationale
- Key assumptions
- MVP scope
- Explicit trade-offs
- "Not Doing" list

## Key Principles

- One question at a time — never a list of questions
- Push back honestly on weak concepts rather than affirming everything
- Surface untested assumptions explicitly
